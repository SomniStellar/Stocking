# 대시보드 차트 개발 정리

## 목적
- Dashboard 차트 개발에 필요한 현재 기준을 한 문서로 정리한다.
- 기능 범위, 데이터 구조, 실패 처리, 개발 순서를 분산 문서 대신 한 번에 확인할 수 있도록 한다.

## 현재 결론
- 차트 1차 범위는 `기본 선형 그래프 + 선택 시점 값 표시`다.
- 차트 1차 구현 라이브러리는 Recharts다.
- benchmark 비교 차트는 benchmark 카드 섹션 바로 아래에 배치한다.
- 기간 버튼은 차트 안의 일부 포인트가 아니라 차트 전체 범위를 바꾸는 토글로 동작한다.
- 즉 `YTD`, `3Y`, `5Y` 중 하나를 누르면 해당 기간 범위 전용 차트로 전체가 전환된다.
- `Brush` 기반 범위 조절과 극점 표시는 후속 단계 검토 항목이다.

## 입력 데이터 기준
### 현재 구현 기준
- 현재 Dashboard 비교 카드 계산은 `Monitor`, `Benchmarks`를 사용한다.
- 그래프 단계 전까지 `Monitor`는 유지한다.
- 현재 차트 준비 단계에서는 `Monitor + Benchmarks` 기반 가공 계층부터 만든다.

### 그래프 단계 확장 기준
- 그래프 단계에서는 `Quotes`, `SeriesCalendar`, `Series`를 추가한다.
- `YTD`는 당해연도 `DAILY` 시계열을 사용한다.
- `3Y`, `5Y`는 최근 5년 `WEEKLY` 시계열을 사용한다.
- 화면은 저장된 범위를 절단해서 사용한다.
- 현재 프로젝트는 거래이력 기반이 아니라 `현재 보유 상태` 기준이므로, Portfolio 시계열은 우선 임시 계산으로 시작한다.

## 차트 데이터 처리 기준
- 차트 UI보다 데이터 가공 계층을 먼저 만든다.
- `DashboardPage`는 기간 선택과 화면 렌더링에 집중한다.
- 차트용 selector 또는 helper가 원본 snapshot을 차트용 데이터로 변환한다.
- 정규화 값은 시트에 저장하지 않고 데이터처리 계층에서 계산한다.
- 계산식은 `(point_value / base_value) * 100`이다.
- 시작점 이전 유효값이 없으면 해당 시리즈는 해당 기간 비교에서 제외한다.
- benchmark 비교 차트는 선택된 기간 하나에 대해서만 시계열을 렌더링한다.
- 현재 Portfolio 시계열은 거래이력 반영값이 아니라 현재 `Holdings` 상태 기준 추정값일 수 있음을 전제로 본다.

## 벤치마크 상태 처리 기준
- `ticker_primary`를 먼저 조회한다.
- primary 실패 시 최대 2회 재시도한다.
- 재시도 후에도 실패하고 fallback이 있으면 fallback으로 전환한다.
- fallback까지 실패하면 `status=failed`로 저장한다.
- `failed` 상태가 되어도 `is_enabled`는 자동으로 변경하지 않는다.
- `failed` 상태의 benchmark는 비교 계산과 차트 렌더링에서 자동 제외한다.
- fallback 사용 시 대체 지표 사용 상태를 표시한다.
- fallback까지 실패하면 오류 캡션을 표시한다.

## 차트 렌더링 기준
- 기간 토글은 차트 전체 범위를 전환하는 역할을 한다.
- benchmark 비교 차트는 선택된 한 기간 범위에 대해 `Portfolio`와 활성 benchmark를 함께 렌더링한다.
- 마우스 오버 값은 선택된 기간 범위 안의 해당 시점 값을 보여준다.
- `failed` benchmark는 차트 선에서 제외하고 상태 캡션만 표시한다.
- Desktop 기준으로 카드 섹션 아래에 자연스럽게 이어지는 레이아웃을 유지한다.

## 상단 포트폴리오 그래프 아이디어
- 상단 포트폴리오 요약 섹터에 `현재 포트폴리오의 기간별 수익금 추이 그래프`를 별도로 두는 안을 후속 검토 항목으로 둔다.
- 이 그래프는 benchmark 비교 차트와 역할이 다르다.
- 상단 그래프는 포트폴리오 자체 흐름에 집중한다.
- 하단 그래프는 benchmark와의 상대 비교에 집중한다.
- 상단 포트폴리오 수익금 그래프는 정확한 시계열이 필요하므로 `Series` 구조 도입 이후 구현하는 것을 기준으로 한다.
- 다만 거래이력 모델이 없는 현재 구조에서는 정확한 과거 수익금 그래프가 아니라 추정 그래프가 될 수 있다.

## 개발 진행 순서
1. 구글시트 현재 템플릿 구조를 먼저 반영한다.
2. `Monitor + Benchmarks` 기준 차트용 데이터 가공 계층을 분리한다.
3. benchmark 상태 처리 로직 `primary -> fallback -> failed`를 반영한다.
4. benchmark 비교 차트를 기간 전환형 구조로 구현한다.
5. 카드 계산값과 차트 계산값을 샘플 데이터로 검산한다.
6. 그래프 단계에서 `Quotes`, `SeriesCalendar`, `Series` 구조를 추가한다.
7. 이후 상단 포트폴리오 수익금 그래프를 검토한다.

## 현재 개발 blocker 여부
- 현재 남은 미결정사항은 대부분 개발 blocker가 아니다.
- 즉시 구현을 막는 구조상 문제는 없다.
- 다만 포트폴리오 계산 검산은 문서 확정 전이라도 개발 중 바로 확인해야 한다.

## 아직 남은 미결정사항
- `KOSPI`, `KOSDAQ` fallback 후보 티커
- fallback 사용 문구 기준
- 오류 캡션 문구 기준
- 디버그 export 플래그 이름 및 위치
- 포트폴리오 계산 결과 검증 기준
